## 1. Pendahuluan & Masalah

*Exploratory Data Analysis* (EDA) adalah proses kritis dalam alur kerja ilmu data modern, bertujuan untuk memahami data melalui visualisasi, pembangunan model tentatif, dan iterasi. Karena sifatnya yang **eksploratif** dan tujuannya yang seringkali **samar** (*vague*), EDA menghadirkan banyak tantangan. Ilmuwan data, terutama pemula atau yang kurang pengalaman, sangat bergantung pada pengetahuan eksternal, seperti contoh *notebook* daring, untuk mendapatkan inspirasi dan petunjuk langkah selanjutnya.

Ketergantungan saat ini pada mesin pencari eksternal (misalnya, Google Search) mengharuskan pengguna beralih konteks antara lingkungan JupyterLab dan peramban web. Pergeseran ini mengganggu alur kerja dan menyulitkan pemahaman contoh kode yang rumit dan tidak relevan. Penelitian sebelumnya mengenai otomasi EDA terbatas pada operasi tingkat rendah (*low-level*) dan kurangnya keterlibatan manusia.

::: tip Solusi yang Diusulkan
**EDAssistant** adalah alat bantu interaktif dan visual yang diimplementasikan sebagai ekstensi **JupyterLab**. Alat ini memfasilitasi EDA dengan kemampuan **pencarian kode *in situ*** (berbasis konteks kode yang sedang dikerjakan) dan **rekomendasi API** berdasarkan repositori *notebook* yang ada, didukung oleh visualisasi interaktif hasil pencarian (*DNA Plot*).
:::

## 2. Metodologi

EDAssistant dibangun di atas tiga komponen utama: Modul *Data Preprocessing and Storage*, *Back-end Analytics Engine*, dan *Front-end Visual Interface*.

### A. Karakterisasi Proses EDA dan Data
1. **Data Collection:** Mengumpulkan **38.581 *notebook*** EDA berkualitas tinggi dari kompetisi Kaggle.
2. **Program Slicing:** *Notebook* mentah dipecah menjadi **EDA Sequences** (segmen kode yang dapat dieksekusi) berdasarkan dependensi variabel dan panggilan fungsi, dengan asumsi urutan EDA berakhir ketika menghasilkan *output* visual atau tekstual.
3. **Program Analysis:** Menganalisis *EDA Sequences* untuk mendapatkan:
    * **API Execution Orders:** Urutan penggunaan API dari *toolkit* ilmu data umum (pandas, scikit-learn, dll.).
    * **Code Keywords:** Kata kunci deskriptif untuk setiap blok kode (menggunakan TF-IDF).
    * **Code EDA Operations:** Menggunakan **LDA Topic Modeling** (dengan kata kunci API) untuk mengidentifikasi empat jenis blok operasi EDA kanonik: *Configuration/Data Preparation*, *Model Exploration/Development*, *Hypothesis Verification/Evaluation*, dan *Output Examination/Visualization*.

### B. Code Search dan API Recommendation
EDAssistant menggunakan model *Deep Learning* **GraphCodeBERT** (yang memanfaatkan *dataflow graph* dalam program) yang disempurnakan (*fine-tuned*) untuk dua tugas hilir:

1. **Search Engine (Pencarian Contoh EDA):**
    * *GraphCodeBERT* di-*fine-tuned* dengan lapisan *mean pooling* untuk menghasilkan **Code Embeddings** (vektor 768 dimensi) dari seluruh *EDA Sequences*.
    * Selama inferensi, *embedding* dari kode pengguna saat ini digunakan untuk menemukan *EDA Sequences* yang paling relevan dalam korpus berdasarkan **kesamaan kosinus**.

2. **API Recommender (Rekomendasi API Berikutnya):**
    * Menambahkan *Dense Layer* setelah lapisan *embedding* *GraphCodeBERT* untuk memprediksi **vektor *one-hot*** (19.453 dimensi) yang merepresentasikan API berikutnya yang paling mungkin digunakan.
    * Model ini dilatih menggunakan *API execution orders* sebagai *ground truth*.

## 3. Detail Pengujian

### Dataset dan Lingkungan
*   **Dataset:** $\mathbf{38.581}$ *notebook* EDA dari Kaggle (dijadikan $\mathbf{236.501}$ *EDA Sequences*).
*   **Lingkungan:** Ekstensi **JupyterLab**.
*   **Peserta Studi Pengguna:** 14 partisipan (ilmuwan data/mahasiswa dengan keahlian data science menengah ke atas).
*   **Tugas:** Dua tugas EDA: 1) Membuat *plotting* (terstruktur); 2) Investigasi data lebih dalam dan membangun model prediksi (terbuka).

### Baseline
*   **Code Search Baseline:** **Doc2Vec** (untuk perbandingan model) dan **Google Search** (untuk perbandingan praktik saat ini dalam studi pengguna).

### Metrik Evaluasi Model
Untuk evaluasi API Recommender, digunakan:
1. **Accuracy (Akurasi):** Rata-rata jumlah API yang diprediksi dengan benar dibagi dengan jumlah API *ground truth* sebenarnya.
2. **Intersection Over Union (IOU):** Untuk menghukum set prediksi yang terlalu besar, dihitung sebagai:
$$
IOU = \frac{|P \cap G|}{|P \cup G|}
$$
Di mana $P$ adalah set API yang diprediksi dan $G$ adalah set API *ground truth*.

## 4. Hasil Eksperimen

### A. Perbandingan Model Search Engine
| Model | Sampel True yang Ditemukan (Top 100) |
| :--- | :--- |
| **GraphCodeBERT** | $\approx 3500$ |
| Doc2Vec | $\approx 250$ |
*   **GraphCodeBERT** secara signifikan mengungguli **Doc2Vec** dalam tugas simulasi pencarian kode, menunjukkan representasi yang unggul untuk *EDA Sequences*.

### B. Perbandingan Model API Recommender
| Model | Accuracy | IOU |
| :--- | :--- | :--- |
| **GraphCodeBERT + Dense Layer (API Recommender)** | **0.507** | **0.399** |
| Doc2Vec | 0.491 | 0.389 |
| GraphCodeBERT Only (Search Engine Top 1) | 0.146 | 0.079 |
*   Model *API Recommender* khusus (dengan *Dense Layer*) menunjukkan kinerja yang jauh lebih baik daripada hanya mengandalkan API dari hasil pencarian teratas (*Search Engine Only*).

### C. Hasil Studi Pengguna (RQ3)

| Indikator Kinerja | Google Search | EDAssistant |
| :--- | :--- | :--- |
| Partisipan yang Berhasil Bangun Model | 6/14 ($\mathbf{43\%}$) | 10/14 ($\mathbf{71\%}$) |
| Waktu Penyelesaian (Menit, M) | 9.8 | 12.9 |
*   EDAssistant secara signifikan meningkatkan kemampuan partisipan untuk menyelesaikan tugas EDA yang terbuka (membangun model ML), mencapai tingkat keberhasilan $\mathbf{71\%}$.
*   Meskipun waktu yang dihabiskan dengan EDAssistant sedikit lebih lama, perbedaannya **tidak signifikan**, menunjukkan bahwa peningkatan efektivitas tidak mengorbankan kecepatan secara substansial.
*   Partisipan memuji pengalaman pencarian **in situ** sebagai lebih *smooth* dan **DNA Plot** (yang memvisualisasikan blok operasi EDA) sebagai intuitif dan membantu pemfilteran.

## 5. Kesimpulan

EDAssistant berhasil memenuhi tujuan desainnya untuk menyediakan dukungan EDA yang terintegrasi dan kontekstual di dalam *computational notebooks*. Melalui pemanfaatan model *GraphCodeBERT* pada korpus *notebook* Kaggle, EDAssistant secara efektif menyediakan **pencarian kode *in situ*** dan **rekomendasi API** yang relevan, didukung oleh visualisasi yang inovatif. Studi pengguna menunjukkan bahwa EDAssistant membantu partisipan menjadi lebih berhasil dalam menyelesaikan tugas EDA yang kompleks (membangun model prediksi) dibandingkan praktik saat ini menggunakan mesin pencari eksternal.

::: info Dampak Praktis
EDAssistant menjembatani kesenjangan antara lingkungan JupyterLab dan pengetahuan kolektif ilmu data. Dukungan *in-context* ini memungkinkan ilmuwan data untuk: (1) **Belajar dan mendapatkan inspirasi** lebih cepat, (2) **Mengurangi gangguan** yang disebabkan oleh beralih konteks pencarian, dan (3) **Meningkatkan probabilitas** keberhasilan dalam mencapai tujuan eksplorasi data yang terbuka dan kompleks.
:::
