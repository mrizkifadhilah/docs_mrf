## ðŸ“š Inti Materi

### 1. Shared Responsibility Model
Keamanan di Cloud terbagi dua:
* **Security OF the Cloud (Tugas Provider):** Keamanan fisik data center, hardware server, dan jaringan global.
* **Security IN the Cloud (Tugas Developer):** Keamanan data pelanggan, enkripsi, manajemen akses, dan *patching* kode aplikasi. Kamu tidak bisa "terima beres".

### 2. Musuh Bebuyutan: OWASP Top 10
OWASP (Open Web Application Security Project) secara berkala merilis daftar 10 risiko keamanan web paling kritis. Dua "selebriti" jahat yang paling sering muncul adalah:

* **SQL Injection (SQLi):**
    Penyerang menyuntikkan perintah database berbahaya lewat input form.
    * *Contoh:* Di kolom login, user mengetik `' OR '1'='1`. Jika tidak diamankan, database mengira ini perintah "Benar" dan memberikan akses admin tanpa password.
    * *Solusi:* Gunakan **Prepared Statements** (jangan gabungkan string query secara manual).

* **Broken Access Control:**
    Kegagalan membatasi apa yang boleh dilakukan user.
    * *Contoh:* User biasa bisa mengakses halaman `/admin` hanya dengan mengetik URL tersebut di browser.

### 3. Prinsip Pertahanan Utama
* **Input Validation & Sanitization:** Jangan pernah percaya input dari user. Semua data yang masuk (dari form, URL, API) harus dianggap "beracun" sampai divalidasi dan dibersihkan.
* **Principle of Least Privilege (Hak Akses Minimum):** Berikan hak akses sekecil mungkin yang dibutuhkan seseorang untuk bekerja.
    * *Contoh:* Aplikasi pencatat stok gudang tidak boleh punya izin untuk menghapus tabel database utama. Jika aplikasi itu diretas, kerusakannya terbatas.

## ðŸ“± Contoh Penerapan: Form Komentar Blog
Bayangkan sebuah blog tanpa keamanan:
1.  **Serangan XSS (Cross-Site Scripting):** Hacker menulis komentar: `<script>curi_cookie()</script>`.
2.  **Dampak:** Setiap kali orang lain membaca komentar itu, *browser* mereka akan menjalankan skrip tersebut dan data mereka dicuri.
3.  **Pertahanan (Sanitization):** Aplikasi secara otomatis mengubah simbol `<` menjadi `&lt;` sebelum menyimpannya, sehingga browser membacanya sebagai teks biasa, bukan kode program.

## ðŸ“– Mini-Glossary
* **OWASP:** Komunitas global yang menetapkan standar keamanan perangkat lunak.
* **SQL Injection:** Teknik peretasan dengan memanipulasi perintah database melalui input aplikasi.
* **Least Privilege:** Prinsip keamanan di mana setiap entitas (user/sistem) hanya diberi izin minimal yang mutlak diperlukan.
* **DevSecOps:** Filosofi pengembangan di mana keamanan (Security) diintegrasikan sejak awal proses Development dan Operations, bukan di akhir.

### 5 Soal Pilihan Ganda (HOTS)

1.  **Analisis:** Jika data pelanggan di database bocor karena developer lupa menutup *port* database ke publik, tetapi server fisiknya aman di data center AWS, siapa yang bertanggung jawab menurut *Shared Responsibility Model*?
    * a. Sepenuhnya AWS karena mereka penyedia layanan.
    * b. Sepenuhnya Developer karena kesalahan konfigurasi keamanan (Security IN the Cloud).
    * c. Pemerintah.
    * d. Hacker.
    * e. Vendor Database.

2.  **Skenario:** Seorang karyawan magang diberi akses "Super Admin" ke seluruh sistem agar "mudah bekerja". Seminggu kemudian, akun laptopnya kena virus dan seluruh database perusahaan terhapus. Prinsip keamanan apa yang dilanggar di sini?
    * a. Defense in Depth.
    * b. Security by Obscurity.
    * c. Principle of Least Privilege.
    * d. Input Validation.
    * e. Cloud Computing.

3.  **Teknis:** Bagaimana cara paling efektif mencegah serangan *SQL Injection* pada kode backend?
    * a. Menggunakan Firewall mahal.
    * b. Menggunakan *Prepared Statements* atau *Parameterized Queries* yang memisahkan kode SQL dari data input.
    * c. Melarang user mengetik tanda kutip.
    * d. Menggunakan database NoSQL saja.
    * e. Menyembunyikan nama database.

4.  **Identifikasi:** Manakah di bawah ini yang merupakan contoh serangan *Broken Access Control* (salah satu OWASP Top 10)?
    * a. Server mati karena kebanjiran traffic (DDoS).
    * b. User A bisa melihat tagihan medis User B hanya dengan mengubah ID di URL browser dari `.../user/A` menjadi `.../user/B`.
    * c. Password user terlalu pendek.
    * d. Aplikasi meminta izin kamera.
    * e. Database lambat diakses.

5.  **Konsep:** Apa tujuan utama dari praktik "Input Sanitization" (Pembersihan Input)?
    * a. Agar data terlihat rapi di database.
    * b. Mencegah input berbahaya (seperti skrip jahat atau karakter SQL) dieksekusi oleh sistem sebagai perintah.
    * c. Menghemat ruang penyimpanan.
    * d. Mempercepat proses login.
    * e. Memastikan user mengetik dengan ejaan yang benar.

### Kunci Jawaban
1.  **b** (Keamanan konfigurasi dan data adalah tanggung jawab *user* cloud).
2.  **c** (Memberikan akses berlebihan meningkatkan dampak risiko jika terjadi peretasan).
3.  **b** (Solusi standar industri untuk menetralkan input berbahaya di SQL).
4.  **b** (Kegagalan sistem memverifikasi apakah User A *berhak* melihat data User B).
5.  **b** (Mengubah input berbahaya menjadi format teks yang aman).
